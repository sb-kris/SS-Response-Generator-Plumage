# Plumage

> Demo data, fully feathered.

Plumage is an internal tool for the SurveySparrow Presales team. It generates persona-driven, contextually-aware, multilingual survey responses and pushes them directly into a SurveySparrow workspace via the public REST API — replacing hours of manual ChatGPT prompting + cleanup before each customer demo.

**Audience:** SurveySparrow Solutions Consultants, SEs, and CSMs. Internal use only.

> **Where it fits.** Plumage is part of **Flock** — a small GTM toolkit for the SurveySparrow team. Its companion tool is **Weave**, a survey builder hosted at <https://surveysparrow-quick-importer-by-sujith.onrender.com/>. The enablement deck that introduces both tools ships with this app at [`/deck`](/deck) (alias [`/flock`](/flock)).

## Status

Phases 1, 2, 3a, 3b, and 3c shipped. Remaining sub-phases of 3 (per-question, custom variables, system metadata, timing, live cost estimator) plus phases 4–9 are stubbed in [Build Spec v2](./BUILD_SPEC.md) and will land incrementally.

### Phase 1 — Setup & auth

- Password-protected access via Next.js middleware (rate-limited login, signed HTTP-only cookie).
- Setup screen with two cards:
  - **SurveySparrow** — region picker (US / EU / UK / AP / CA / IN / ME), optional workspace nickname, API key. Test Connection probes `GET /v3/surveys?limit=1`.
  - **LLM** — provider toggle (Anthropic / OpenAI), API key, separate model selectors for personas vs responses, with hover tooltips covering cost / speed / use case. Test Connection sends a 1-token probe.
- Persistent header connection-status bar with relative-time freshness ("tested 2m ago"), a "stale" hint after 10 minutes, and the workspace nickname inline next to the SurveySparrow pill.
- Light + dark mode (defaults to dark).
- Toasts for connection success / failure.
- Sign-out confirmation dialog clears in-memory state and the IndexedDB question cache.

### Phase 2 — Survey selection & question loading

- New top-nav with **Setup** + **Generate** links. Generate is unblocked once both connections are healthy; otherwise the page renders a friendly "connect first" CTA.
- `/generate` lists every survey from the connected workspace via `POST /api/surveysparrow/surveys` (walks all pages of `/v3/surveys`).
- Searchable survey combobox built on `cmdk` — search by name or type. Each row shows the survey type as a badge, the relative modified date, and an "archived" hint when applicable.
- On selection, `POST /api/surveysparrow/questions` walks `/v3/questions?survey_id=…` and renders the result as an accordion: question text, type badge, required flag, choices (for MCQ/dropdown), scale points (for rating-style), and a numeric scale chip when only `scale` is set.
- Per-survey question payloads cache in IndexedDB (via `idb-keyval`) for 30 minutes. Cache keys are namespaced by `region + last-6-chars of API key` so two SS workspaces in the same browser don't share entries. **Only payloads cache — never API keys.** A "Refresh" button invalidates the cache for the selected survey and re-fetches.
- Loading skeletons throughout; error states with a retry button surface the upstream error verbatim.
- The question preview header surfaces a bucket summary ("23 questions · 5 open-text · 8 ratings · 6 multiple choice · 4 other") via the new question-type registry.
- Question types that aren't generated by the LLM (welcome / thank-you / file upload / voice / video / contact form) get a subtle "Skipped during generation" badge. The registry's `answerable` flag is the single source of truth.

### Phase 3a — Configure step foundation + Demo Profiles

- `/generate` now runs as a 4-step wizard: **Select survey → Configure → Preview → Generate & push**. Steps 3 + 4 are locked until Phases 6 + 7. The current step is persisted to sessionStorage so an accidental refresh doesn't reset progress.
- The **Configure** step uses a three-column layout: left rail of section anchors with active-section highlighting via `IntersectionObserver`, center pane with the form sections, sticky right rail with a preliminary cost estimator. Future sub-phases (themes, personas, language, per-question, variables, metadata, timing) appear as locked anchors and dashed placeholder cards so users can see what's coming.
- The first real section, **Use case context**, is a 2,000-character textarea with a live counter and a "what to include" hint card.
- **Demo Profile system** (full CRUD): saves the entire configuration draft (every Phase 3 sub-phase, even the ones still landing) as a named profile in IndexedDB. Auto-suffix on name collisions ("Foo" → "Foo (2)"). Import / export as JSON for sharing across teammates.
- The **ProfileBar** at the top of Configure shows the loaded profile (with an "Unsaved changes" badge when the draft diverges) and exposes Save / Load / Save as new / Reset / Manage all profiles. **`Cmd/Ctrl + S`** opens the save dialog.
- The configuration draft is persisted to **sessionStorage** so an in-progress configuration survives refresh but a tab close starts clean. Saved profiles persist in IndexedDB across sessions.
- New top-nav link: **Profiles** (`/profiles`) — list / load / duplicate / export / delete saved profiles, plus "Import JSON" to bring profiles in from teammates.
- Sign-out clears all in-memory state, IndexedDB profiles + question cache, and the sessionStorage draft.

The full DemoProfile schema in [`lib/profiles/types.ts`](lib/profiles/types.ts) is designed up front for every sub-phase of Phase 3 (themes, persona distribution, language distribution, per-question configs, custom variables, system metadata, time range). Sub-phases 3b–3g just plug into their respective fields — no schema migrations needed.

### Phase 3b — Themes & Persona Distribution

- **Themes** ([`ThemesSection`](components/generate/configure/ThemesSection.tsx)) — tag-style input with Enter-to-add, max 8 themes, inline-editable labels (click to rename), 0–100 weight slider + numeric input per theme, delete button. Suggestions are survey-type-aware via [`lib/themes/suggestions.ts`](lib/themes/suggestions.ts) (NPS / CSAT / CES / default buckets), with 6 chips visible by default and a "Show more" toggle. Duplicate adds and over-cap inputs surface friendly toasts. Empty state guides the user toward 2–5 themes.
- **Persona distribution** ([`PersonasSection`](components/generate/configure/PersonasSection.tsx)) — three color-coded sliders (Promoters / Passives / Detractors, default 60/25/15) wired through a lock-aware [`rebalance()` helper](lib/profiles/persona-distribution.ts) so changing one slider proportionally adjusts the others while preserving the sum-to-100 invariant. Per-slider Lock toggles freeze a value during auto-balance. Numeric inputs alongside each slider stay synced. Smart presets (Healthy / NPS-focused / Recovery / Polarized) animate the sliders into place via `requestAnimationFrame` with cubic ease-out; the matching preset highlights as "active". A `recharts` donut on the right reflects the live distribution. The active preset card highlights when the current values exactly match a preset.

### Phase 3c — Language distribution (multilingual support)

- **Languages section** ([`LanguagesSection`](components/generate/configure/LanguagesSection.tsx)) — per-language sliders with locks + numeric inputs, sum-to-100 enforcement via the generic [`rebalanceToTotal()`](lib/utils/sum-to-total.ts) helper, "Auto-balance" button, and a sum badge that turns warning when the total drifts off 100. English is always present and never deletable; removing other languages redistributes their weight proportionally to the remaining unlocked rows (falling back to English when everything else is locked).
- **Add language picker** ([`AddLanguagePopover`](components/generate/configure/AddLanguagePopover.tsx)) — Popover + `cmdk` Command with searchable list of all 13 languages. Shows native name + English name + ISO code, marks already-added languages as "Added", single-click adds at 0% weight (user explicitly allocates).
- **Country preview chips** ([`CountryPreviewChips`](components/generate/configure/CountryPreviewChips.tsx)) — under each language slider, top 3 countries surface as compact `🇲🇽 MX 35%` chips. A "+N more" toggle expands a full breakdown grid plus an explanatory note.
- **Geography registry** ([`lib/utils/language-geography.ts`](lib/utils/language-geography.ts)) — the data backbone. 13 languages × multiple countries each, with weights, real city names, IANA timezones, phone formats, and per-country Faker locales. Country flag emojis are computed from the ISO 3166-1 alpha-2 code at runtime via regional indicators. Module-load assertion verifies country weights sum to 100 per language (warning logged if a typo slips in). Ready for Phase 4 to consume directly for persona metadata generation.
- **Defensive validation** — on mount, the section strips any unsupported language codes from imported / restored drafts and dumps their weight into English with a toast warning. This protects against profiles exported from a future Plumage version that knows more languages.
- **Soft suggestions** — adding a 6th+ language surfaces a non-blocking "may dilute realism" hint via toast.

## Tech stack

- **Next.js 15** (App Router) on **React 19** with TypeScript strict mode.
- **Tailwind CSS** + **shadcn/ui** components (default style, Slate base, indigo accent).
- **Zustand** for in-memory client state (no persist middleware — API keys vanish on refresh).
- **next-themes** for theme switching.
- **sonner** for toasts.
- Edge-Runtime-safe HMAC-SHA256 cookie signing via Web Crypto.

## Getting started

### Prerequisites

- Node.js 20+ (we test on 24).
- pnpm 10+ (`npm i -g pnpm`).

### Setup

```bash
pnpm install
cp .env.example .env.local
# edit .env.local — set APP_PASSWORD and AUTH_SECRET
openssl rand -hex 32   # use this output for AUTH_SECRET
pnpm dev
```

Open <http://localhost:3000>. You'll be redirected to `/login` — enter your `APP_PASSWORD`, then test both connections on the setup screen.

### Environment variables

| Variable | Required | Description |
|----------|:--------:|-------------|
| `APP_PASSWORD` | ✅ | Shared password for the team. Anyone with this can access Plumage. Rotate when the team changes. |
| `AUTH_SECRET` | ✅ | 32+ byte random string used to sign the auth cookie. Generate with `openssl rand -hex 32`. |
| `MONTHLY_COST_CAP_USD` |   | Soft cost-cap warning threshold (default 100). Phase 3+ will surface this in the cost estimator. |

API keys (SurveySparrow, Anthropic, OpenAI) are **never** stored as env vars — they are entered in the UI and held in browser memory only. Refresh the tab to clear them.

## Deploying to Vercel

1. Push this repo to GitHub.
2. Import into Vercel — it auto-detects Next.js.
3. Set `APP_PASSWORD` and `AUTH_SECRET` in the Vercel project's Environment Variables.
4. Deploy. The middleware runs on the Edge by default — cookie signing uses Web Crypto, so it's compatible.

> **Caveat:** the login rate-limiter is in-memory per serverless instance. For an internal tool with a strong password, this is fine; if we ever expose Plumage more widely, swap in Upstash Redis (10 lines in `lib/auth/rate-limit.ts`).

## Security model

- **Auth.** Single shared `APP_PASSWORD`. On success, the server signs `<timestamp>.<HMAC-SHA256>` and sets it as an HTTP-only, SameSite=Lax cookie (Secure in production). The middleware verifies this on every non-public request.
- **Rate limiting.** 5 login attempts per IP per minute (in-memory bucket).
- **API key handling.**
  - Never written to localStorage, IndexedDB, env files, server logs, or the auth cookie.
  - Sent to our Next.js API routes per request and used once before being discarded.
  - All `/api/*` responses set `Cache-Control: no-store` (configured in `next.config.ts`).
- **CORS.** Same-origin only (Next.js default).
